## Internal Linking Rules

1. **Vertical Links (Pillar ↔ Cluster)**
   * Every cluster article links up to its local pillar.
   * Every local pillar links down to all its cluster articles.
   * Every local pillar links up to the Global Main Pillar.
   * The Global Main Pillar links down to all 8 local pillars.

2. **Horizontal Links (Sideways)**
   * Cluster articles in different buckets can link to each other when topically relevant.
   * Use the `/posts/{slug}` format for all internal links.
   * Only link to articles that are **published** or **working locally** (ready for publish).
   * Future/unpublished articles must be mentioned as **plain text only** — no hyperlinks.

3. **No-Duplicate Rule**
   * Database per Service Pattern Explained lives in Bucket 2 only.
   * Bucket 4 (Design Patterns) links to it but does not duplicate it.
   * Microservices Architecture Diagram Explained lives in Bucket 1 as a cluster and in Bucket 2 as the local pillar. It is a single article serving both roles.

4. **Link Format**
   * Internal: `[Article Title](/posts/{slug})`
   * No trailing slash.
   * No absolute domain (`https://alnashiri.dev`).
   * No relative paths (`../`).

5. **When to Add Links**
   * Add links only when the target article is published or working locally.
   * When a "Next Coming" article gets published, revisit all articles that should link to it and add the links.

6. **Frontmatter Category and Tag Rules**
   * Each article must have exactly one `category` — use the bucket's category.
   * Do not assign two categories to one article, even if it serves dual bucket roles.
   * `tags` must describe article-specific topics, not pillar relationships or planning labels.
   * Pillar/cluster relationships are expressed through internal links, not through tags.
   * Do not use broad tags like `System Design` — prefer `Software Architecture` or `Distributed Systems` as appropriate.
